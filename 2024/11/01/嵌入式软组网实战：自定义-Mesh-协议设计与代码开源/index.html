嵌入式软组网实战：自定义SoftMesh协议设计与代码开源

1 前言

1.1 机遇

一次星闪体验官的机会，接触到了Hi3863这款搭载WiFi、蓝牙、星闪三种通讯能力于一体的开发板，由于星闪技术那时候还没有类似Mesh组网的能力，于是想自己填补这方面的空白。由于星闪技术还比较新，不确定性比较大，于是想先从WiFi的Mesh组网复现下手，主要这方面有很多的参考资料可以学习和借鉴，先拿WiFi做成功的概率比较大，WiFi做成功之后再移植到星闪上。但是后续由于星闪体验官活动的结束和个人精力的原因，只是勉强实现了WiFi的Mesh组网，没能移植到星闪上。如果对这部分感兴趣的开发者，欢迎在此基础上移植到星闪或者其他厂商不自带组网方案的开发板上。

1.2 挑战

尽管WiFi的Mesh组网实现有参考的案例，但是在真正想参考时发现没有真正意义上可以参考的代码。在做这个项目之初，发掘到能够参考的案例有：IEEE 802.11s协议标准、ESP32 Mesh方案、ESP8266的easyMesh方案。但是这些参考的方案的参考价值有各自的局限性。

IEEE 802.11s作为标准协议，虽然规范完善，但实际应用受限。大多数支持WiFi功能的嵌入式芯片并不支持该标准。从Open80211s的开源实现和应用场景来看，该标准主要面向运行Linux系统的家用路由器设备，难以在资源受限的嵌入式设备上实现。

ESP32 Mesh方案虽然提供了开源代码，但仅开放了使用接口，核心的mesh协议实现被封装在静态库中。这种实现方式虽然保证了稳定性，但也限制了开发者根据特定场景进行定制化开发的可能性。不过，其官方文档对实现原理的详细说明仍具有重要的参考价值。

ESP8266的easyMesh方案虽然完全开源，但由于与ESP8266底层代码深度耦合，且该芯片已较为老旧，其参考价值主要体现在组网思路和上层接口设计方面。

1.3 解决方案

由于并没有现有很好的方案能够给我用在Hi3863组网问题上，于是我决定自己开发一个名为SoftMesh的组网协议，并且将代码完全开源到了Gitee和Github上，希望能让更多的物联网设备具备组网能力。该协议在设计时重点关注以下几个方面：

  1. 高度可移植性：协议架构设计采用模块化思想，可以轻松移植到不同的嵌入式芯片平台，同时支持灵活的扩展和定制化修改。

  2. 多协议兼容：通过抽象通信接口层，实现了对不同通信方式（如WiFi、蓝牙、星闪等）的兼容支持，为异构网络组网提供了技术基础。

  3. 去中心化架构：采用完全分布式设计，无需中心节点，每个节点都具有自主管理能力。当网络出现断链时，节点可以智能选择新的父节点，确保网络的可靠性和自愈能力。

这种设计不仅解决了现有方案的局限性，也为物联网领域的组网应用提供了一个灵活、可靠的解决方案。

1.4 需求定义（实现效果）

每个节点使用统一的Mesh协议代码，节点之间应该是平等的，节点的角色应该由Mesh协议自行协商。使用Mesh协议时，用户只需要定义Mesh的网络ID，节点启动后会自动和同ID的节点进行组网，组网的过程应该由Mesh协议自主控制，对用户时无感的。组网成功后，节点能发送消息给Mesh网络内的任意节点。

2 SoftMesh 协议简介：一套轻量自组织网络协议

2.1 系统总览

SoftMesh的系统架构图如下图所示，从上向下分别是应用接口层、网络状态管理层、路由与数据传输层、硬件抽象层。对于一个Mesh网络来说，最核心的任务可以简化为两部分，一个是组网，一个是路由。组网是实现节点之间物理上通过无线连接起来，而路由是为了能够让节点知道目标节点的位置，实现正常的网络通讯。组网对应在网络状态管理层实现，路由在路由与数据传输层实现。

2.2 应用接口层

应用接口层是SoftMesh协议的顶层，为上层应用提供统一的接口，支持多种应用场景。该层提供了以下核心接口：

  1. 网络初始化接口

    int mesh_init(const char *ssid, const char *password);

    该接口用于初始化Mesh网络，需要提供网络SSID和密码。成功返回0，失败返回-1。这是使用Mesh网络的第一个必须调用的接口。

  2. 点对点通信接口

    int mesh_send_data(const char *dest_mac, const char *data);

    用于向指定MAC地址的节点发送数据。目前支持最大490字节的数据包，适合大多数物联网应用场景。

  3. 广播通信接口

    int mesh_broadcast(const char *data);

    用于向网络中所有节点广播数据，同样支持最大490字节的数据包。适用于需要向所有节点发送控制命令或状态更新的场景。

  4. 数据接收接口

    int mesh_recv_data(char *src_mac, char *data);

    非阻塞式接收数据接口，可以获取发送节点的MAC地址和接收到的数据。该接口设计为非阻塞式，适合在事件循环中使用。

  5. 网络状态查询接口

    int mesh_network_connected(void);

    用于查询当前节点是否已连接到Mesh网络。返回1表示已连接，0表示未连接。可用于网络状态监控和故障诊断。

这些接口设计遵循了简单易用的原则，同时保持了足够的灵活性。开发者可以基于这些接口快速构建各种物联网应用，如智能家居控制、工业监控、环境监测等场景。

下面是一个典型的使用示例，展示了如何使用这些接口实现一个简单的Mesh网络节点：

void test_api_task(void)
{
    LOG("test_api_task run...\n");
    // 初始化Mesh网络，设置SSID和密码
    if(mesh_init("FsrMesh", "12345678") != 0) {
        LOG("mesh_init failed.\n");
    } else {
        LOG("mesh_init successfully.\n");
    }

    while (1)
    {
        // 检查网络连接状态
        if (mesh_network_connected() == 0) {
            LOG("Mesh network not connected.\n");
            osDelay(100);
            continue;
        }

        osDelay(100);
        // 广播数据
        // if (mesh_broadcast("Hello, Mesh!") != 0) {
        //     LOG("mesh_broadcast failed.\n");
        // } else {
        //     LOG("mesh_broadcast successfully.\n");
        // }
        
        // 接收数据
        char src_mac[7] = {0};
        src_mac[6] = '\0';
        char data[512] = {0};
        if(mesh_recv_data(src_mac, data) == 0) {
            LOG("Received data from client: %s, MAC: %s\n", data, src_mac);
            
            // 收到数据后，向发送方回复消息
            LOG("say hi to mac: %s\n", src_mac);
            mesh_send_data(src_mac, "Hi, Node!");
        } else {
            LOG("no data recv.\n");
            continue;
        }
    }
}

这个示例展示了以下关键功能：

  1. 网络初始化：使用mesh_init初始化Mesh网络，设置SSID和密码。

  2. 网络状态监控：通过mesh_network_connected持续监控网络连接状态。

  3. 数据收发：

    • 使用mesh_recv_data接收其他节点发送的数据
    • 使用mesh_send_data向特定节点发送回复消息
    • 使用mesh_broadcast向网络中所有节点广播消息（当前被注释）

  4. 错误处理：对每个操作都进行了错误检查，确保网络操作的可靠性。

这个示例实现了一个简单的”回显”功能，当收到其他节点的消息时，会回复一个”Hi, Node!”消息。这种模式可以用于：

  • 网络连通性测试
  • 节点发现
  • 简单的双向通信

如果将代码中的广播部分取消注释：

if (mesh_broadcast("Hello, Mesh!") != 0) {
    LOG("mesh_broadcast failed.\n");
} else {
    LOG("mesh_broadcast successfully.\n");
}

则节点会定期向网络中所有其他节点广播”Hello, Mesh!”消息。这种广播模式适用于：

  • 网络状态同步
  • 全局控制命令下发
  • 节点状态通告

2.3 网络状态管理层

网络状态管理层是SoftMesh协议的核心层，负责实现网络的自组织功能。该层采用有限状态机（FSM）设计，通过精确的状态转换控制，实现网络的自组织。有限状态机的状态转换图如下所示。

由于WiFi在STA（Station）模式下只能连接一个AP（Access Point），因此SoftMesh网络天然形成了一个树状拓扑结构，这种结构具有以下特点：

  • 层次清晰，便于路由管理
  • 支持多跳通信
  • 具备自愈能力，当某个节点断开时，其子节点可以重新寻找父节点

2.4 路由与数据传输层

路由与数据传输层负责路由和数据传输任务，在这一次最核心的部分是路由表的数据结构和使用方式，而数据传输和转发在路由实现之后只需要进行简单的逻辑控制即可。完整的构建该协议的路由表需要使用三个数据结构，分别是哈希表、邻接表和父节点数组。

哈希表用于记录MAC地址和索引的关系，MAC地址是节点在一个Mesh网络的唯一标识，但是如果直接拿MAC地址做索引会十分的不方便，所以单独使用了一个哈希表记录MAC地址和索引的映射关系，如下图示例A1B2C3的对应索引是2后续将用索引构建路由表。

邻接表用于记录节点与节点之间的连通关系，类似key-value的结构，每一个节点记录其邻居节点（包括父节点和子节点），如下图的案例所示节点1的邻居节点分别是节点0、节点5和节点6。

仅仅使用邻接表只能表征节点之间连通关系，并不能构建树状图的父子关系，因此还需要单独使用一个父节点数组记录每个节点的父节点。该数组更多的作用是用来溯源路径，比如0号节点想发消息给6号节点，可以再父节点数组递归找6号节点的父节点，就可以找到6→1→0的发送路径。0号节点就知道把数据转发给1号节点处理就行了。

2.5 测试层

在开发的过程中，每一层的代码目录都有三个文件夹，分别是inc、src、test这三个目录，其中test目录就是单独测试该层功能的代码，同时在每层的.c源码文件里面，可以寻找ENABLE_LOG 宏定义，将其打开可以看到更多该部分代码的运行过程日志文件，感兴趣的开发者可以通过单独运行test代码和打开日志宏更加细致的了解该协议运行机制。

由于目前个人精力有限，各层只是介绍了部分核心的内容，后续有时间会单独介绍各层的具体实现细节。

3 项目展示

目前SoftMesh适配的芯片是hi3863，如果需要复现该项目，购买使用该芯片的开发板或者自己的设计的开发板都行。因为该项目没有使用任何外设接口，只是用了WiFi的功能，因此理论上兼容任何使用hi3863的开发板。目前也只适配了hi3863芯片，理论上只要具备STA和AP共存能力的开发板，都能很容易的移植适配本组网协议。

3.1 基础验证实验

基础实验验证的内容分别是：自组网、广播、单播、根节点下线自恢复。该实验使用了4个节点，其中3个节点运行的是普通的接收代码，另外1个节点还加入了广播的代码。硬件接线如下图所示，首先上电三个普通节点，验证节点的自组网功能呢，然后再上电广播节点，验证节点的广播能力，各个节点收到广播之后，会向广播节点回应一个消息，验证节点的单播能力。最后会找到这个Mesh网络的根节点，直接断电下线，验证Mesh网络的自恢复能力。

4 经验总结与展望

4.1 总结

做这个项目的初心本来是想实现星闪Mesh组网功能，但是单单复现学习WiFi Mesh的过程就耗费了1个月的时间。尽管在复现之初已经梳理清楚了思路和原理，但是将理论转为实践总会遇到各种各样的不可预测的一些问题和一开始没有考虑到的细节。也正是因为一开始梳理清楚的思路和需求，做好了版本迭代记录，让工程向着有序和收敛的方向发展，而不是向着代码爆炸的“屎山”方向奔溃。不管怎么样，这个项目自认为算是我本科阶段做的较为完整的项目之一，融合了工程、数据结构、算法的知识为一体。但也不可否认的是，在只做了一个月时间的情况下，这个项目也不可能做到尽善尽美到可以到商业化使用的地步。内部的很多细节还有待优化的地方，例如在后期赶工期的时候，为了快速实现组网和数据传输的效果，直接定死了数据包的大小，也没有去实现大数据包的分割机制，类似这样的优化问题有很多，我也意识到这些细节都是一个无底洞，我目前也不可能投入那么多时间精力去维护这些细节。幸运的是，我成功的在星闪体验官的活动结束前实现了WiFi Mesh的组网和通讯功能，打造好了Soft Mesh的项目框架。不幸的是，我没有时间把这个工程移植到星闪上，去验证我的框架。

当然，我也在社区上看到了一些别的小伙伴在Mesh网络上的尝试，有的使用beacon类似蓝牙那样的泛洪的思路构建Mesh，也有人把Mesh的根节点，中转节点、普通节点的角色写死，不同的角色使用不同的代码，由人工设计不同节点在不同场景下的角色。这样做虽然省去了节点自主协商等步骤，但这样做显然也把Mesh组网的精华部分都抛弃了，灵活性大大降低。
